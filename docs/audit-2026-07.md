# LiftLog full-app audit — July 2026

A senior-engineer review of every workflow in the app: the data layer
(`src/data`), IndexedDB layer, all components, and the Cloudflare worker.
Baseline at audit time: **230/230 tests passing, clean build**. The codebase is
in genuinely good shape — the pure-engine architecture (every piece of
intelligence is a pure function of `(program, TrainingSnapshot)`) has held up
across many iterations, sync v2's merge semantics are sound, and test coverage
of the data layer is strong. The findings below are ranked by user impact.

## Fixed in this audit (see the accompanying commit)

### Bugs

1. **Dashboard "X of N done" counted one-off workouts** (`Dashboard.tsx`).
   Quick workouts (dayId −2) and shared workouts (−1) landed in
   `completedDayIds`, so logging a one-off inflated the weekly progress tally
   ("3 of 4 done" after two program days + a quick session). Now filtered to
   program day ids. *(Data quality, UX)*

2. **Quick/shared/orphaned sessions couldn't be edited from History**
   (`App.tsx`). "Edit Session" looks the day up in the current program; for
   one-off sessions (negative dayIds) and sessions from days removed by a
   re-plan, the lookup failed and the app silently bounced to the dashboard.
   WorkoutView's edit mode already renders every logged exercise regardless of
   the day design, so App now synthesizes a stub day and those sessions are
   fully editable. *(UX, reliability)*

3. **Worker 500s leaked internals** (`worker/index.ts`). Error responses
   included `err.message` and the full stack trace — table names, query
   shapes. Now logged server-side only. *(Security)*

### Hardening

4. **No size cap on the pushed program** (`worker/sync.ts`). `validatePush`
   capped sessions/set logs/plan bytes but stored `program_json` unbounded.
   Added a 1 MB cap (same as the plan document). *(Reliability, abuse)*

5. **Expired `user_sessions` rows accumulated forever** (`worker/auth.ts`).
   Dead tokens are now swept opportunistically on each sign-in. *(Hygiene)*

### Performance

6. **Hot-path localStorage re-parsing** (`exercises.ts`, `programStore.ts`).
   `getExerciseMeta`/`profileFor` are called per exercise inside candidate
   ranking, the planner, and muscle resolution — and each call re-parsed the
   entire metadata blob (and often the library + tombstones) from
   localStorage. Added parse caches **keyed on the raw stored string**, so
   every write path — including direct localStorage writes in tests and the
   account-switch wipe — stays cache-coherent with no explicit invalidation.
   Candidate ranking and plan generation now do O(1) parses instead of
   O(exercises × factors).

### Code simplicity

7. **Three copies of the exercise picker.** `DayEditView` and
   `QuickWorkoutView` each carried an inline duplicate of the
   search-library-then-create picker that `AddExercisePanel` already
   implements. The panel now takes `confirmLabel` and a `persistent` mode
   (multi-add flows), and both views use it — ~200 lines deleted, one place to
   maintain the "never mint a duplicate identity" rule.

8. **Dead code removed.** `program.ts` `getExerciseName` (shadowed by the
   real one in `programStore.ts`, zero importers) and `plan.ts`
   `blockLengthWeeks` (zero importers). Fixed the `bumpSets` comment in
   `planner.ts`, which described the opposite of what the code does (adds go
   to the *lightest* qualifying slot — correct behavior, wrong comment).

### Exercise intelligence

9. **`sameLift` false positives on generic names** (`substitution.ts`).
   The same-movement check was token-subset-only, so a custom exercise named
   "Squat" blocked *every* squat-family name from being suggested — including
   "Bulgarian Split Squat", which is a Lunge-pattern movement, not the same
   lift. `sameLift` now compares profiles: a known, differing movement pattern
   overrides the name match. Unit-tested.

10. **CLAUDE.md brought up to date.** It predated milestone 21 (quick
    workouts, warm-up sets, sharing, glossary, admin merges, mid-workout add);
    the file map, View union, localStorage keys, IDB schema and decisions now
    match the code. *(Maintainability — this file is the context every future
    session works from.)*

## Recommended next (not done here — need product decisions or larger changes)

Ranked by expected payoff:

1. ✅ **Surface sync failures** — *implemented right after the audit*:
   `data/syncStatus.ts` (status store: ok/offline/auth/error + pushPending +
   persisted last-success timestamp), a dashboard banner when sync is failing
   (with a re-login link on expired sessions), an always-visible status line
   in Settings, and automatic push retry on the 60 s background tick, window
   focus, and the browser `online` event. Original recommendation kept below
   for context.

   **Surface sync failures** *(Reliability — highest-impact open item).*
   `pushSync` failures land in `console.error` and nowhere else. A user whose
   pushes silently fail for a week loses that week on their next device
   switch. Suggest: a tiny sync-status dot (last push OK / failing / offline)
   on the Settings screen or dashboard header, plus a retry-on-focus. The
   plumbing (`sync()` already centralizes pushes in App) makes this cheap.

2. **RPE/RIR logging** (already in the roadmap). The deload trigger still
   can't distinguish "grinding at RPE 10" from "easy reps". One optional field
   per set; fully backwards-compatible.

3. **Quick-workout draft collisions** *(UX edge).* All quick workouts share
   dayId −2, so an abandoned quick-workout draft (< 12 h old) restores into
   the *next* quick workout, merging stale sets into a fresh session. The
   Discard banner mitigates it. Options: stamp drafts with a session nonce, or
   offer "Resume your unfinished quick workout?" on the quick-setup screen.

4. **Warm-up-aware "last time" line** *(polish).* The ExerciseCard "last
   time" line and recommendation history read working sets only (correct),
   but History's set numbering counts warm-ups in `setNumber` while the label
   says "Warm-up" — harmless, slightly odd ("Set 2" is the first working set
   in the row list). Cosmetic renumber at render time if it bothers anyone.

5. **Worker-side tests** (roadmap). `validatePush`, the session-doc upsert
   SQL, and the pull fallback chain (user rows → app tables → legacy JSON)
   are the highest-value targets; `vitest-pool-workers` is the tool.

6. **Bundle size** *(minor).* 410 KB minified / 122 KB gzip is fine for a
   PWA shell that's precached, but `PlanSetupView` + `planner` + wizard CSS
   (~60 KB source) could lazy-load via `React.lazy` if it ever matters.

7. **`/api/auth/me` endpoint has no client callers.** Harmless (useful for
   debugging); either use it for a session-validity check on focus (would
   catch the "cookie present but session expired server-side" state, where
   the app currently shows the UI but every sync 401s silently — related to
   item 1) or delete it.

8. **Two `nameToDef` maps** (analytics.ts, substitution.ts) — same
   name→catalog-def lookup built twice. Cosmetic duplication; fold into one
   export if either grows logic.

## Verified-sound (things checked that need no change)

- **Sync v2 merge semantics**: per-document LWW with tombstones is correctly
  implemented on both ends; local-only sessions can't be dropped by a pull;
  every set-rewriting path calls `touchSession`. The 60 s background pull +
  focus pull + startup sequence compose correctly with the account-switch
  wipe (`ensureLocalDataOwner` runs first).
- **Empty-program safeguards**: both directions (client refuses to adopt an
  empty server program; worker refuses to store an empty program) plus
  `reconcileActiveProgram` self-heal from the active block.
- **Week math**: Monday-anchored, whole-day rounding (DST-safe), local-time
  date parsing everywhere (`parsePlanDate`, `parseLocalDate`,
  `dateInputToTimestamp`).
- **Warm-up exclusion**: `buildSnapshot` keeps warm-ups out of
  `setsBySession`, so every analytical consumer (metrics, coach, progress,
  recommendations, heatmap, retrospectives) is warm-up-clean by construction;
  history renders `allSetsBySession`.
- **XSS surfaces**: the QR modal's `dangerouslySetInnerHTML` renders only
  library-generated SVG from an app-encoded payload; share-import decoding
  validates shape and clamps all values/lengths.
- **Admin API**: role-checked server-side on every route, all writes audited,
  merge cycle-guard correct.
- **IDB discipline**: reads and writes in separate transactions; multi-record
  writes queue synchronously and await `txDone` — consistent with the
  documented auto-commit rules.
